> **선택 2시간 코스**

## 이번 장에서 할 것

1. 저장소를 써보면서 바꾸고 싶은 부분을 정리합니다.
2. bkit `/plan` 스킬로 요구사항을 계획 문서로 만듭니다.
3. PDCA 흐름을 이해합니다.

---

## 1단계. 바꾸고 싶은 것 정리하기

저장소를 열어보고, 로컬에서 실행해보면서 "이 부분은 이렇게 바꾸고 싶다"는 생각을 모읍니다.

Claude에게 이렇게 물어보면서 정리해도 됩니다.

```txt
이 저장소를 내 라이프 위키로 쓰려면 어떤 부분을 바꿔야 해?
내가 원하는 건 [원하는 것]인데, 지금 이 구조로 가능한지 알려줘.
```

---

## 2단계. `/plan` 스킬 실행하기

채팅창에 입력합니다.

```txt
/plan
```

bkit이 "어떤 기능을 기획할까요?" 하고 물어보면 이렇게 답합니다.

```txt
내 라이프 위키를 이렇게 수정하고 싶어.
[바꾸고 싶은 것들을 자유롭게 적습니다]
```

bkit이 계획 문서를 자동으로 생성합니다.

<details>
<summary>생성된 계획 문서를 확인하고 싶다면</summary>

```txt
방금 만든 Plan 문서 어디 있어? 내용도 보여줘.
```

마음에 안 드는 부분이 있으면 이렇게 말합니다.

```txt
Plan 문서에서 [이 부분]을 [이렇게] 바꿔줘.
```

계획 문서는 처음부터 완벽하지 않아도 됩니다. 방향만 잡혀 있으면 충분합니다.

</details>

---

## 3단계. PDCA 흐름 이해하기

bkit은 아래 흐름으로 작업을 진행합니다.

```txt
Plan  → 무엇을 만들지 계획
Do    → 계획한 대로 만들기
Check → 제대로 됐는지 확인
Act   → 다음에 고칠 점 반영
```

이 장이 **Plan** 단계입니다. 다음 장부터 **Do** 단계로 넘어갑니다.

<details>
<summary>왜 기획을 먼저 하나요?</summary>

계획 없이 바로 만들면 이런 상황이 생깁니다.

- 어느 순간 "지금 뭘 하고 있었지?" 하고 길을 잃습니다.
- 만들다가 방향이 바뀌어서 앞 작업이 낭비됩니다.
- 다 됐을 때 무엇을 확인해야 하는지 모릅니다.

계획을 먼저 잡으면 이런 상황을 줄일 수 있습니다. 계획은 한 번 정하면 변경할 수 없는 규칙이 아닙니다. 진행하면서 바꾸면 됩니다.

</details>

---

## 이번 장에서 확인한 것

1. 써보면서 바꾸고 싶은 부분을 발견했습니다.
2. `/plan` 스킬로 요구사항을 계획 문서로 만들었습니다.
3. PDCA는 계획 → 만들기 → 확인 → 반영의 흐름입니다.
