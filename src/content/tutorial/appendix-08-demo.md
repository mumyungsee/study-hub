## 이 가이드를 쓰는 상황

아래가 준비된 상태에서 시작합니다.

- Antigravity 설치됨
- bkit 설치됨
- git 설치됨
- GitHub CLI(`gh`) 설치됨, 로그인 상태
- GitHub 계정 있음 (로그인 상태)
- Vercel 계정 있음 (GitHub 연결됨)
- 내 컴퓨터에 작업 폴더(`life-wiki`) 만들어져 있음
- Antigravity에서 그 폴더가 열려 있음

## 0:00 — Fork + Clone + 폴더 열기 (7분)

### Fork — 강사 레포를 내 GitHub로 복사

브라우저에서 강사 레포에 들어갑니다.

```txt
[강사 레포 URL]
```

오른쪽 위 **`Fork`** 버튼 → 내 계정 선택 → **`Create fork`**

주소가 바뀐 것을 확인합니다.

```txt
Fork 전: github.com/강사아이디/레포이름
Fork 후: github.com/내-아이디/레포이름
```

### Clone — 내 컴퓨터로 내려받기

Antigravity 채팅창에 입력합니다.

```txt
내 GitHub에 있는 [레포이름] 저장소를
지금 열린 life-wiki 폴더 안으로 clone 해줘.
```

Clone이 끝나면 왼쪽 파일 탐색기에 파일들이 채워집니다.

<details>
<summary>GitHub 로그인 오류가 났다면</summary>

```txt
GitHub CLI 로그인이 필요한지 확인하고, 필요하면 로그인 과정 안내해줘.
```

</details>

## 0:07 — 레포 파악 (3분)

채팅창에 입력합니다.

```txt
이 레포가 어떤 구조인지 설명해줘.
어떤 파일들이 있고, 주요 기능이 뭔지 한눈에 보여줘.
```

구조를 파악한 다음, 내가 고치고 싶은 부분을 생각합니다.

아직 아무것도 고치지 않습니다. 먼저 무엇을 바꿀지 정합니다.

## 0:10 — Plan: 기획하기 (10분)

### `/pdca plan` 실행

채팅창에 입력합니다.

```txt
/pdca plan
```

bkit이 "어떤 기능을 기획할까요?" 라고 물어봅니다. 내가 원하는 것을 자유롭게 적습니다.

아래 예시 중 내 상황에 맞는 것을 골라 입력합니다.

---

### 요구사항 예시 — 내 상황에 맞게 골라 쓰세요

**예시 A. 운동·식사 일지 중심**

```txt
이 레포를 내 건강 기록 공간으로 만들고 싶어.

- 하루 회고에 식사 일지(아침/점심/저녁) 항목 추가
- 운동 기록 섹션 추가 (종류, 시간, 강도)
- 블로그 소재 수집은 빼고 대신 건강 인사이트 메모 공간 추가
- 주간 요약에 컨디션 점수 평균이 나오면 좋겠어
```

**예시 B. 강의 콘텐츠 제작자**

```txt
이 레포를 강의 소재 수집 공간으로 쓰고 싶어.

- 블로그 소재 대신 강의 소재(아이디어, 사례, 예시) 수집 항목으로 변경
- 하루 일과 중 '강의 작업' 시간을 별도로 기록
- 코칭 질문 대신 "이 주제로 강의를 만든다면 어떤 순서로 설명할까?" 질문으로 변경
- 주간 회고에 '이번 주 강의 준비 진행도' 항목 추가
```

**예시 C. 다른 회고 방법론 적용**

```txt
지금은 자유로운 일기 형식인데, KPT 방법론으로 회고 구조를 바꾸고 싶어.

KPT는:
- Keep: 잘 됐으니 계속할 것
- Problem: 문제가 된 것
- Try: 다음에 시도할 것

이 세 항목으로 하루 회고 섹션을 재구성해줘.
주간 회고도 한 주 KPT로 정리하게 바꿔줘.
```

**예시 D. 마음챙김·감정 기록 중심**

```txt
코칭 질문보다는 감정 기록에 집중한 공간으로 만들고 싶어.

- 하루 중 가장 강하게 느낀 감정 1~2개를 기록하는 항목 추가
- "오늘 나를 힘들게 한 생각이 있다면?" 하는 인지치료 기반 자기점검 질문 추가
- 감사한 것 3가지 기록 (gratitude journal)
- 코칭 세션 항목은 제거
```

---

bkit이 계획 문서를 자동으로 만들어줍니다. 내용을 확인하고 마음에 들면 계속 진행합니다.

### 계획 문서 커밋

```txt
변경된 파일 확인하고, 커밋 메시지는 "plan: 라이프 위키 커스텀 계획"으로 커밋해줘.
push는 아직 하지 마.
```

## 0:20 — Do: 수정하기 (20분)

계획이 잡혔으면 Claude에게 직접 수정을 요청합니다.

### 순서대로 하나씩

한 번에 다 요청하지 말고, **하나씩** 수정하고 확인하면서 진행합니다.

```txt
Plan 문서 기준으로 첫 번째 항목부터 수정해줘.
하나 끝나면 알려줘, 그러면 다음 단계 말할게.
```

### 중간에 로컬 미리보기로 확인

```txt
로컬 미리보기 실행해줘.
```

브라우저에서 `http://localhost:4321` 접속해서 바뀐 부분이 실제로 어떻게 보이는지 확인합니다.

확인 후 종료:

```txt
로컬 서버 종료해줘.
```

### Do 커밋

원하는 수정이 됐으면 체크포인트를 남깁니다.

```txt
변경된 파일 확인하고, 커밋 메시지는 "do: [오늘 한 작업 요약]"으로 커밋해줘.
push는 아직 하지 마.
```

## 0:40 — Check + 배포 (10분)

### Check: 계획대로 됐는지 확인

```txt
/pdca analyze
```

계획한 것 중 완료된 것과 안 된 것이 나뉘어 표시됩니다.

```txt
✓ 식사 일지 항목 추가 — 완료
✓ 운동 기록 섹션 추가 — 완료
✗ 주간 요약 컨디션 평균 — 미완료
```

미완료 항목은 지금 바로 수정하거나, 다음 사이클로 넘깁니다.

### Act: 다음 사이클을 위한 메모

```txt
이번 작업에서 잘 된 것 한 가지, 다음에 이어서 할 것 한 가지를 정리해줘.
CLAUDE.md에 간단히 메모로 추가해줘.
```

### Push + 배포

```txt
지금까지 변경된 파일을 확인하고,
커밋 메시지는 "act: 첫 번째 커스텀 완료"로 커밋한 다음 GitHub에 push해줘.
```

Push가 끝나면 Vercel이 자동으로 사이트를 다시 빌드합니다.

브라우저에서 Vercel 대시보드를 엽니다.

```txt
https://vercel.com/dashboard
```

`Building` → `Ready`로 바뀌면 완료입니다. 내 사이트 URL로 접속해서 확인합니다.
