## 이번 장에서 할 것

7장에서 만든 계획 문서를 바탕으로 실제 수정을 진행합니다.

1. 계획 문서를 Claude에게 보여주고 수정을 시작합니다.
2. 수정 결과를 로컬 미리보기로 확인합니다.
3. 마음에 안 드는 부분은 바로 수정 요청합니다.

---

## 1단계. 수정 시작하기

채팅창에 입력합니다.

```txt
Plan 문서 보고 수정 시작해줘.
하나씩 순서대로 진행해줘.
```

Claude가 계획 문서를 읽고 첫 번째 항목부터 수정을 시작합니다.

수정이 끝나면 무엇을 바꿨는지 요약해줍니다. 확인 후 다음 항목으로 넘어갑니다.

<details>
<summary>잘 되는 요청 방식</summary>

Claude에게 요청할 때 세 가지를 포함하면 더 잘 됩니다.

1. **무엇을** 바꾸고 싶은지
2. **어느 파일**과 관련된 건지
3. **다 되면** 무엇을 확인해야 하는지

예시:

```txt
CLAUDE.md에서 사이트 이름 부분을 "내 라이프 위키"로 바꿔줘.
바꾼 다음 파일 내용 전체를 보여줘.
```

막연하게 "뭔가 바꿔줘"보다 구체적일수록 Claude가 잘 도와줍니다.

</details>

<details>
<summary>Claude가 한 작업을 되돌리고 싶다면</summary>

```txt
방금 한 것 되돌려줘.
```

또는 더 앞으로 돌아가고 싶으면:

```txt
git status 보여줘. 변경된 파일 목록 확인하고 싶어.
```

```txt
[파일 이름] 원래대로 되돌려줘.
```

</details>

---

## 2단계. 로컬 미리보기로 확인하기

수정이 어느 정도 됐으면 실제로 어떻게 보이는지 확인합니다.

```txt
로컬 미리보기 실행해줘.
```

브라우저에서 `http://localhost:4321`로 접속합니다.

확인이 끝나면 종료합니다.

```txt
로컬 서버 종료해줘.
```

---

## 3단계. 추가 수정하기

미리보기를 보면서 더 바꾸고 싶은 부분이 생겼으면 바로 요청합니다.

```txt
[이 부분]이 마음에 안 들어. [이렇게] 바꿔줘.
```

수정 → 미리보기 → 추가 수정을 반복합니다. 이게 PDCA의 Do 단계입니다.

<details>
<summary>PDCA가 뭔지 궁금하다면</summary>

bkit은 작업을 네 단계로 나눠서 진행합니다.

```txt
Plan  → 무엇을 바꿀지 계획 (7장)
Do    → 실제로 수정 (지금 이 장)
Check → 제대로 됐는지 확인
Act   → 다음에 개선할 것 반영
```

지금 이 장이 **Do** 단계입니다. 수정하고 미리보기로 확인하는 것 자체가 Check까지 포함됩니다.

완벽하게 다 바꾸지 않아도 됩니다. 오늘 하고 싶은 것만 하고 Push하면 됩니다. 다음에 또 이 사이클을 돌리면 됩니다.

</details>

---

## 4단계. 수정 내용 커밋하기

수정이 마음에 들면 지금 상태를 커밋해둡니다.

```txt
변경된 파일 확인하고, 바꾼 내용을 요약해서 커밋 메시지 만들어줘.
"do: " 로 시작하게 해줘. push는 아직 하지 마.
```

<details>
<summary>중간에 커밋하는 이유</summary>

수정이 한참 진행된 뒤 "아, 이 부분은 되돌리고 싶은데"라는 상황이 생깁니다. 커밋이 없으면 처음부터 다시 해야 합니다.

중간중간 커밋해두면 원하는 시점으로 돌아갈 수 있습니다.

```txt
[커밋 메시지] 시점으로 되돌려줘.
```

</details>

---

## 이번 장에서 확인한 것

1. 계획 문서를 바탕으로 Claude와 실제 수정을 진행했습니다.
2. 로컬 미리보기로 결과를 확인했습니다.
3. 수정 내용을 커밋해서 체크포인트를 만들었습니다.
